03 / 09
Patrones de ingeniería

Decisiones de diseño que mantienen el código predecible a escala.

/01

Endpoints dinámicos

El frontend no llama URLs específicas. Despacha a un dispatcher universal que importa el módulo, instancia la clase y ejecuta el método indicado.

# POST /em/
module=sales_man
package=sales_control
attr=SalesControl
mname=export_rutero_input
# → DExecution.execute_module(...)
/02

Auditoría in-row estandarizada

Cada modelo transaccional lleva los mismos campos de estado: quién lo cargó, verificó, aprobó, procesó o anuló — y cuándo. Reemplaza el soft-delete formal y deja trazabilidad completa.

cargado_010 · verificado_045
aprobado_050 · procesado_060
anulado_040
# + *_por_gecos / _fecha / _hora
/03

Multi-database explícito

Sin routers automáticos: cada query usa .using('acosales') o la DB que corresponda. Módulos enteros (stats, GPS, RTDC) escriben a bases dedicadas. Aislamiento real por empresa del grupo.

Model.objects.using('acosales')
    .filter(anulado_040=False)
    .select_related('cliente')
/04

Sin try/except defensivo

Los métodos invocados vía em están envueltos por @track_error. Las excepciones llegan a Sentry y se traducen al frontend automáticamente — el código de negocio se queda limpio.

@track_error
def export_rutero_input(self, query_dict):
    # lógica de dominio, sin guardas
    return {'ok': True, ...}
